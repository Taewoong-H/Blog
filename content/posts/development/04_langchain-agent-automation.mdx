## 왜 LangChain을 공부하게 됐나

요즘 LangChain이라는 이름을 자주 듣는다. "이건 쓸 줄 알아야 한다"는 얘기도 있고, 관련 교육 과정도 눈에 띈다. 어떤 회사 구인서에서는 아예 LangChain 사용 경험을 역량 항목에 넣어두기도 한다. 그러다 보니 자연스럽게 이런 생각이 들었다.

> "에이전트 자동화를 하려면, 나도 이걸 꼭 알아야 하는 건가?"

그래서 직접 파봤다. 결론부터 말하면 **LangChain이 필수는 아니었다.** 그건 여러 선택지 중 하나일 뿐이고, 정작 중요한 건 그 아래에 깔린 "에이전트가 대체 뭔지"였다. 이 글은 내가 그 질문에서 출발해 밑바닥부터 감을 잡은 순서를 그대로 정리한 것이다.

## 1. 에이전트 = LLM + 툴 + 루프

LLM 하나만 놓고 보면 할 수 있는 건 **텍스트를 받아 텍스트를 뱉는 것** 딱 하나다. 날씨도 못 보고, 파일도 못 읽고, API도 못 부른다.

여기서 **툴(tool)** 이 등장한다. 모델에게 "너 이런 함수 쓸 수 있어"라고 알려주면, 모델은 직접 답하는 대신 "나 `getWeather('서울')` 부르고 싶어"라고 말할 수 있다. 그런데 초보자가 거의 다 놓치는 핵심이 있다.

> 모델은 함수를 **직접 실행하지 않는다.** 그냥 "부르고 싶다"고 *말만* 한다. 실제로 실행해서 결과를 다시 모델에게 넘기는 건 **내 코드**가 해야 한다.

그리고 이걸 **반복(loop)** 으로 감싼 게 바로 에이전트다.

```js
let messages = [{ role: "user", content: "서울 날씨 알려줘" }]

while (true) {
  const res = await model.call({ messages, tools: [getWeather] })

  if (res.type === "text") return res.text          // 끝

  const result = await getWeather(res.toolInput)     // 실행은 내 코드가
  messages.push(res, { role: "tool", content: result }) // 결과 되먹임 → 반복
}
```

**이 루프가 에이전트 자동화의 전부다.** LangChain이든 LangGraph든 전부 이 루프를 얼마나 감싸주느냐의 차이일 뿐이다.

- **공식 SDK** — HTTP를 감싼 얇은 껍데기. 인증·재시도·타입은 해주지만 **루프는 내가 짠다.** 원리를 통제하고 싶을 때.
- **에이전트 SDK (Vercel AI SDK)** — 루프가 사라진다. `generateText({ tools, maxSteps })` 한 줄이면 툴 호출 루프를 대신 돌려주고 UI 스트리밍까지 된다. 프론트엔드에서 AI 기능 붙일 때 가장 자연스럽다.
- **LangChain** — 메모리, RAG, 수십 개 연동까지 미리 만들어진 부품 상자. 조립은 빠르지만 추상화가 두꺼워 "안이 안 보인다"는 불만이 유명하다.
- **LangGraph** — 흐름을 **그래프(상태 기계)** 로 직접 그린다. 분기·재시도·사람 승인이 섞인 복잡한 에이전트에 강하다.

## 3. 워크플로우 vs 에이전트 — 순서를 누가 정하는가

여기서 내가 크게 헷갈렸던 지점. "스크랩 → 차트 → 분석 → LLM 종합"처럼 **순서를 내가 고정**한 건 사실 에이전트가 아니라 **워크플로우(파이프라인)** 에 가깝다. 차이는 딱 하나다.

- **워크플로우** — 내가 코드로 순서를 못 박는다. LLM은 그냥 한 단계.
- **에이전트** — LLM에게 툴과 목표만 주면, **모델이 그때그때 판단**해서 어떤 툴을 어떤 순서로 쓸지 스스로 정한다.

그래서 결론:

> 순서가 매일 똑같으면 → **그냥 워크플로우(스크립트)가 낫다.** 에이전트도 LangChain도 필요 없다.
> 상황을 보고 "다음에 뭘 할지"를 매번 다르게 판단해야 할 때만 → **에이전트가 필요하다.**

에이전트 프레임워크부터 깔았다가 정작 만든 게 고정 순서 파이프라인이라 프레임워크가 짐만 되는 경우가 정말 많다. **"이거 순서가 고정인가, 모델이 판단해야 하나?"** 를 항상 먼저 묻자.

## 4. MCP는 경쟁자가 아니라 다른 층이다

MCP를 다뤄본 사람이라면 "그럼 MCP랑은 뭐가 다르지?"가 자연스럽게 나온다. 핵심 반전은 이거다.

> **MCP는 LangChain·SDK와 경쟁하는 물건이 아니다. 아예 다른 층이다.**

- **프레임워크 / SDK** = *지휘자*. 루프 자체를 담당.
- **MCP** = *툴을 꽂는 표준 규격*. 지휘자가 툴을 어떻게 찾고 호출하는지를 표준화한 것. USB-C 같은 것.

앞의 루프에서 "도구 실행" 한 칸을 떠올려 보자. MCP 없이는 그 자리에서 내가 짠 로컬 함수를 부르고, MCP가 있으면 그 자리에서 MCP 서버로 호출을 보낸다. **루프는 그대로다.** 그래서 MCP 없이도 에이전트를 만들 수 있고, 에이전트를 안 만들어도 MCP를 쓸 수 있다(Claude Desktop이 이미 에이전트니까). 둘은 완전히 다른 축이다.

## 5. 두 개의 자리 — 쓰는 사람과 만드는 사람

"그럼 이건 누가 쓰는 건가?"도 오래 헷갈렸는데, **자리가 두 개**여서 그랬다.

- **자리 ① — 이미 만들어진 에이전트를 쓰는 사람.** Claude Desktop·ChatGPT는 그 자체가 에이전트다. 루프가 내장돼 있어 코드 한 줄 없이 MCP 서버만 꽂아 쓴다. 개발자가 아니어도 된다.
- **자리 ② — 자기 제품 안에 에이전트를 만드는 사람.** 내 앱·내 UI·내 툴로 직접 만든다. 여기서 SDK·LangGraph가 등장한다. 이게 개발자의 일이다.

챗봇 서비스로 그리면 이렇게 나뉜다: **프론트엔드(챗 UI)** 와 **백엔드**로 나뉘고, 백엔드 안에서 에이전트 루프(LangGraph 등)가 돌며 LLM API를 호출하고 툴을 쓴다. 그 전체를 API로 프론트에 연결한다. 그리고 툴 자리에는 내가 만든 MCP 서버가 그대로 꽂힌다.

## 6. 코드로 보는 차이 — 직접 루프 vs LangGraph

같은 에이전트를 두 방식으로 짜보면 프레임워크가 뭘 대신 해주는지 확 드러난다. 먼저 **직접 루프**:

```python
def run_agent_manual(user_msg: str) -> str:
    messages = [{"role": "user", "content": user_msg}]
    while True:                              # ← 이 루프가 에이전트의 심장
        res = client.chat.completions.create(
            model="gpt-4o", messages=messages, tools=TOOL_SCHEMA)
        msg = res.choices[0].message
        messages.append(msg)
        if not msg.tool_calls:               # 툴 안 부르면 → 최종 답변
            return msg.content
        for call in msg.tool_calls:          # 부른 툴을 내 코드가 실행
            result = TOOLS[call.function.name](**json.loads(call.function.arguments))
            messages.append({"role": "tool", "tool_call_id": call.id, "content": result})
```

같은 걸 **LangGraph**로:

```python
from langgraph.prebuilt import create_react_agent

agent = create_react_agent(model, tools=[scrape_news, fetch_chart, analyze])
result = agent.invoke({"messages": [("user", "오늘 삼성전자 장 예측해줘")]})
```

`while` 루프, 툴 디스패치, `messages` 관리가 **통째로 사라졌다.** 이게 프레임워크가 대신 해준다는 것의 실체다. 대신 그 루프가 눈앞에서 사라지니, "왜 여기서 멈췄지?"를 디버깅할 때 안이 안 보인다. 그래서 처음엔 직접 루프를 한 번 손으로 짜보는 게 좋다.

LangGraph의 *진짜* 값은 `create_react_agent` 같은 기성품이 아니라, 흐름이 복잡해질 때 노드와 엣지로 그래프를 직접 그리는 부분이다.

```python
builder.add_node("call_model", call_model)
builder.add_node("run_tools", run_tools)
builder.add_node("human_review", human_review)     # 사람 승인 노드
builder.add_conditional_edges("call_model", route) # 조건 분기
builder.add_edge("run_tools", "call_model")        # 툴 결과 → 다시 모델 (루프)
```

노드는 단계, 엣지는 흐름. 이 그래프를 내가 설계하는 게 LangGraph의 핵심이다.

## 마무리 — 그래서 LangChain, 알아야 할까

처음의 질문("나도 꼭 알아야 하나?")으로 돌아오면, 이제 답이 분명해졌다. **LangChain·LangGraph는 "반드시 배워야 하는 필수"가 아니라, 에이전트를 만드는 여러 방법 중 하나다.** 하지만 이번에 직접 파보면서 얻은 건 프레임워크 사용법 자체가 아니었다. 그보다 **에이전트가 어떤 원리로 도는지** 를 이해하게 된 게 진짜 소득이었다.

LLM에 툴을 쥐여주고 루프를 돌린다는 것, 그 루프를 직접 짤 수도 있고 LangGraph 같은 프레임워크에 맡길 수도 있다는 것, 그리고 그 선택이 "흐름이 얼마나 복잡한가"에 달려 있다는 것. 이 감이 생기니 프레임워크는 그저 도구로 보였다.

여기서 한 발 더 나아갈 그림도 보인다. 이 에이전트의 "툴" 자리에 **MCP 서버**를 꽂으면, 내가 만든 도구들을 표준 규격으로 붙여 얼마든지 확장할 수 있다. 한 번 만든 MCP 서버는 내 커스텀 에이전트에도, Claude Desktop 같은 기성 에이전트에도 그대로 재사용된다. 다음에는 백엔드에 MCP를 실제로 연결해 보면 좋은 경험이 될 것 같다.

결국 전부 "LLM + 툴 + 루프" 라는 하나의 아이디어 위에 얹혀 있었다. 원리를 이해하고 만드느냐, 정해진 대로 따라 만드느냐. 나온 서비스는 비슷해 보여도, 그 둘은 분명히 다르다고 생각한다.
